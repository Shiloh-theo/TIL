## 1. Why Migrate from Twilio to Meta WhatsApp Cloud API

### The Problem with Twilio Sandbox
Twilio's WhatsApp sandbox uses a **shared number**. Every user who wants to interact with your bot must first send a specific "join" code to the sandbox number — a manual onboarding step that breaks real-world usability. For a product like a NAFDAC verification bot meant for everyday Nigerians, that friction is a dealbreaker.

### What Meta WhatsApp Cloud API Solves
- Your bot runs on **your own WhatsApp number**
- Anyone can message it directly — zero onboarding friction
- Free to use (pay per conversation beyond free tier)
- Full support for Nigerian numbers
- Official, stable API maintained by Meta

### The Migration Decision
This is an example of **engineering judgment** — recognising that a technical limitation (shared sandbox) creates a product limitation (broken UX) and making the decision to migrate even when it means more setup work. That thinking is what separates engineers from coders.

## 2. How Meta WhatsApp Cloud API Works

### The Webhook Flow
```
User sends WhatsApp message
        ↓
Meta servers receive it
        ↓
Meta POST request → your webhook endpoint (e.g. /webhook)
        ↓
Your Express server processes the message
        ↓
Your server calls Meta API to send a reply
        ↓
User receives response on WhatsApp
```

### Webhook Verification (One-time Setup)
When you register a webhook URL with Meta, they send a GET request with a `hub.challenge` parameter to verify you own the server. Your endpoint must return the challenge value:

```javascript
app.get('/webhook', (req, res) => {
  const mode = req.query['hub.mode'];
  const token = req.query['hub.verify_token'];
  const challenge = req.query['hub.challenge'];

  if (mode === 'subscribe' && token === process.env.WEBHOOK_VERIFY_TOKEN) {
    res.status(200).send(challenge); // ✅ verified
  } else {
    res.sendStatus(403);
  }
});
```

### Receiving Messages
```javascript
app.post('/webhook', (req, res) => {
  const body = req.body;
  const message = body.entry?.[0]?.changes?.[0]?.value?.messages?.[0];

  if (message) {
    const from = message.from;      // sender's phone number
    const text = message.text?.body; // message content
    // → route to your handler
  }

  res.sendStatus(200); // always return 200 quickly or Meta retries
});
```

### Sending a Reply

await axios.post(

## 3. 9ja Checkr API — NAFDAC Verification

Instead of scraping nafdac.gov.ng directly (fragile, breaks when the site changes), v2 uses the **9ja Checkr API** — a stable third-party service that wraps NAFDAC data.

- Free tier: 300 verifications/month
- Returns product name, manufacturer, registration status
- Much more reliable than direct scraping

## 4. Claude Vision for OCR on Product Labels

The bot accepts **photo messages** — a user can send a photo of a product label and the bot extracts the NAFDAC number using Claude's Vision API, then verifies it automatically. This is a full pipeline:

```
Photo received via WhatsApp
        ↓
Download image from Meta servers
        ↓
Send image to Claude Vision API
        ↓
Extract NAFDAC number from image
        ↓
Query 9ja Checkr with extracted number
        ↓
Reply with verification result
```

This is OCR applied in a real product context — directly connecting the OCR learning from earlier this week.

## 6. Key Lesson — node_modules in Git
The repo currently has `node_modules` committed — this is bad practice.
- Bloats the repo with thousands of files
- These files are auto-generated by `npm install` and should never be version controlled
- Fix: add `node_modules/` to `.gitignore`, then run:
```bash
git rm -r --cached node_modules
git commit -m "remove node_modules from version control"
```
