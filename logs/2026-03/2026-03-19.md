## 1. JPA & Hibernate

**JPA (Java Persistence API)** is a specification — an API that defines how Java applications should interact with relational databases. It is not an implementation, just a contract.

**Hibernate** is the most popular implementation of JPA. When you add Hibernate to your Spring Boot project, it fulfills the JPA contract and handles the actual database communication under the hood.

Think of it like this: JPA is the interface, Hibernate is the class that implements it.

---

## 2. DDL Auto

`spring.jpa.hibernate.ddl-auto` controls how Hibernate manages your database schema relative to your entity classes.

Common values:
- `create` — drops and recreates schema on every startup (dev only)
- `update` — updates schema to match entities without dropping data
- `validate` — checks schema matches entities, throws error if not
- `none` — does nothing (production safe)

## 3. PostgreSQL Dialect

```yaml
spring:
  jpa:
    properties:
      hibernate:
        dialect: org.hibernate.dialect.PostgreSQLDialect
```

The dialect tells Hibernate which *flavour* of SQL to generate. Different databases (PostgreSQL, MySQL, Oracle) have different SQL syntax. Setting the dialect ensures the generated SQL is compatible with PostgreSQL specifically — correct data types, correct syntax, correct functions.

## 4. StringBuilder

`StringBuilder` is a mutable sequence of characters in Java. Unlike `String` (which is immutable — every concatenation creates a new object), `StringBuilder` modifies the same object in memory, making it far more efficient for building strings in loops or across multiple operations.

```java
StringBuilder sb = new StringBuilder();
sb.append("Hello");
sb.append(" World");
String result = sb.toString(); // "Hello World"
```

Use `StringBuilder` when concatenating strings inside loops. Use `String` for simple, one-off values.

## 5. Spring Beans — Singleton Scope & Method-Level Creation

**Beans are singletons by default in Spring** — one instance shared across the entire application context.

**Key insight learned today:** When a bean is defined at the class level (e.g. as a field), there is a risk of state being shared and overridden across different requests or usages. Beans should always be created at the **method level** inside `@Configuration` classes to ensure proper lifecycle management and avoid unintended state sharing.

```java
@Configuration
public class AppConfig {

    // ✅ CORRECT — method level bean
    @Bean
    public MyService myService() {
        return new MyService();
    }
}
```

## 6. DTOs + HTTP Status Codes Combined

**DTO (Data Transfer Object)** — a plain object used to carry data between layers. It decouples your internal entity model from what you expose through the API.

Best practice in Spring Boot: always wrap your response in a DTO and pair it with a meaningful HTTP status.

```java
@GetMapping("/user/{id}")
public ResponseEntity<UserDTO> getUser(@PathVariable Long id) {
    UserDTO user = userService.findById(id);
    return ResponseEntity.ok(user); // 200 OK
}

// For creation:
return ResponseEntity.status(HttpStatus.CREATED).body(newUserDTO); // 201 Created

// For not found:
return ResponseEntity.status(HttpStatus.NOT_FOUND).build(); // 404
```

This pattern is what separates junior code from production-grade code.

## 7. Database Referencing (Foreign Keys)

Database referencing = one column in a table pointing to a primary key in another table. This is fundamentally a **foreign key relationship**.

In JPA, this is expressed using:
```java
@ManyToOne
@JoinColumn(name = "account_id") // the foreign key column
private Account account;
```

This means the `Transaction` entity holds a reference to an `Account` entity. In the database, the `transactions` table has an `account_id` column that points to the `id` column of the `accounts` table. A row in `transactions` *depends on* and *links to* a row in `accounts`.

This is foundational for the **Double-Spend Ledger project** — transactions will reference accounts, accounts will reference users.

## 8. OCR — Lossy vs Lossless Compression

**JPEG (lossy compression):** Permanently discards pixel data to reduce file size. Once compressed, that data is gone. Means the image fed to Tesseract has less information = potentially lower OCR accuracy.

**PNG (lossless compression):** Retains every pixel. File size is larger but image quality is fully preserved. This is why PDFBox renders PDF pages as PNG at 300 DPI before passing to Tesseract — maximum pixel data = maximum OCR accuracy.

**Rule of thumb:** Always use PNG when preparing images for OCR processing.

**Tesseract recap:**
- Written in C++, maintained by Google
- `Tess4J` is the Java library wrapper that lets you call Tesseract from the JVM
- Trained data files (`eng.traineddata`) contain years of character recognition training — curves, bends, angles mapped to characters
