## 1. @Version Annotation — Optimistic Locking in JPA

### The Problem It Solves
When multiple users or threads try to modify the same database record simultaneously, without protection the last write wins — silently overwriting someone else's changes. In a financial system this is catastrophic. Two concurrent transactions could both read a balance of ₦10,000, both debit ₦8,000, and both succeed — leaving the account at ₦2,000 instead of going negative as it should.

### How @Version Works
```java
@Entity
public class Wallet {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Version
    private Integer version;  // JPA manages this automatically

    private BigDecimal balance;
}
```

When a record is read, the `version` value is captured. When an update is attempted, JPA adds a `WHERE version = ?` clause to the SQL. If another transaction updated the record between your read and your write, the version number will have changed — the update affects 0 rows, JPA throws `OptimisticLockException`, and the transaction is rolled back.

**Key insight:** Optimistic Locking assumes conflicts are rare and only checks at commit time. It does not lock the row at read time — meaning no blocking, no waiting, high concurrency. This is exactly right for a fintech wallet system.

### Optimistic vs Pessimistic Locking
| Approach | How | Best For |
|---|---|---|
| Optimistic | Check version at write time, rollback on conflict | High read, low write conflict scenarios |
| Pessimistic | Lock row at read time (SELECT FOR UPDATE) | High conflict, critical sections |

For the Double-Spend Ledger — Optimistic Locking is the correct choice.

## 2. BigDecimal — Why It Matters in Financial Systems

### The Problem with double and float
```java
double a = 0.1 + 0.2;
System.out.println(a); // 0.30000000000000004 ❌
```

`double` and `float` use binary floating point representation. They cannot precisely represent most decimal fractions. In a financial system, this means rounding errors accumulate — and rounding errors in money are bugs that cost real money.

### BigDecimal is the Solution
```java
BigDecimal a = new BigDecimal("0.1");
BigDecimal b = new BigDecimal("0.2");
System.out.println(a.add(b)); // 0.3 ✅
```

**Critical rule:** Always initialise BigDecimal from a String, never from a double:
```java
// ❌ WRONG — inherits the floating point imprecision
new BigDecimal(0.1)  // gives 0.1000000000000000055511151231257827021181583404541015625

// ✅ CORRECT — exact decimal value
new BigDecimal("0.1")
```

### Common BigDecimal Operations
```java
BigDecimal balance = new BigDecimal("10000.00");
BigDecimal debit = new BigDecimal("3500.50");

// Arithmetic
BigDecimal newBalance = balance.subtract(debit);   // 6499.50
BigDecimal total = balance.add(debit);              // 13500.50

// Comparison — never use .equals() for value comparison
balance.compareTo(debit) > 0   // balance > debit ✅
balance.equals(debit)          // compares scale too, unreliable ❌

// Rounding for display/storage
balance.setScale(2, RoundingMode.HALF_UP)
```

### In the Double-Spend Ledger
Every monetary field — balance, transaction amount, fees — must be `BigDecimal`. The `@Column` mapping in PostgreSQL maps to `NUMERIC` or `DECIMAL` type.

```java
@Column(precision = 19, scale = 4)
private BigDecimal balance;
```
